1 Potential Limitations to Application of the SWB Code {#potential_limitations}
2 ===============================================================================
3 
4 The original concept behind the SWB code was to allow for the spatial
5 distribution of groundwater recharge to be quickly and easily
6 calculated
7 on the basis of readily available data and a standardized set of
8 parameters (Dripps, 2003). The version of the SWB model
9 described here retains most of the features that made the original
10 model
11 attractive from the standpoint of practical application
12 (Dripps, 2003; Dripps and Bradbury, 2007). Despite the possible
13 limitations given below, the SWB model approach is capable of
14 generating
15 reasonable annual or monthly mean groundwater recharge estimates at
16 the
17 scale of a small catchment (Dripps and Bradbury, 2007). In order to do
18 so,
19 however, the user will have to upscale the daily results offered by
20 the
21 SWB model and average or filter the results over a larger area. The
22 relative spatial variability and pattern of recharge between
23 catchments
24 should also be of great value, particularly if these estimates can be
25 corroborated with recharge estimates generated from streamgages or
26 observation wells. Comparing SWB-calculated recharge estimates to
27 those
28 estimated from streamflow records, or from a groundwater model
29 calibrated to stream fluxes is recommended.
30 
31 As with any numerical model, the burden is on the user to preprocess
32 the
33 input grids in the most appropriate manner. If the user has not done
34 this, then SWB will generally halt after giving the user a description
35 of the problem it detects for an input grid. Although the SWB code can
36 certainly be applied using only available data and a standard set of
37 curve numbers, it would be prudent to treat the results with caution,
38 as
39 one should with any model output. In addition, certain underlying
40 theoretical limitations should be kept in mind when interpreting SWB
41 model output. These limitations are discussed below.
42 
43 **Runoff routing.** The inclusion of overland-flow routing in the code
44 ensures that runoff from an upslope grid cell has one or more
45 opportunities to contribute to infiltration in the cells that are
46 downslope from it. However, all runoff from a cell is assumed to
47 infiltrate in downslope cells or be routed out of the model domain on
48 the same day in which it originated as rainfall or snowmelt.
49 
50 In addition, once water is routed to a closed surface depression and
51 evapotranspiration and soil-moisture demands are met, the only loss
52 mechanism is recharge. This results in cases where maximum recharge
53 values of hundreds or thousands of inches per year are calculated.
54 These
55 extremely high values are unrealistic and likely result from surface
56 storage of water not being accounted for. The code described here
57 allows
58 the user to enter a maximum recharge rate for each land-cover and
59 soil-group combination. This feature offers a way to restrict the
60 estimated recharge values to a more reasonable range; however, the
61 rejected recharge is nonetheless removed from the model domain on the
62 same day in which it originated as precipitation or snowmelt.
63 
64 **Groundwater/surface-water interaction.** Interactions between
65 surface-water and groundwater features are not simulated in the SWB
66 code
67 and could not be without significantly increasing the complexity of
68 the
69 model. In locations where the water table is beneath the bottom of the
70 root-zone, the SWB code should be capable of producing reasonable
71 annual
72 or monthly values. The depth from the bottom of the root zone to the
73 top
74 of the water table is not considered in the estimation of recharge;
75 there may be a significant time of travel through the unsaturated
76 zone.
77 Coupling the SWB code with an unsaturated-zone code that could route
78 water to the water table, such as the MODFLOW UZF Package
79 (Niswonger and others, 2006), would be one way to address this
80 limitation.
81 
82 In areas with wetlands, springs, lakes, or other landscape features
83 where the water table is close to the land surface, the SWB code can
84 be
85 expected to perform poorly; there is currently no provision for
86 recharge
87 rejection via saturation excess other than by specifying a maximum
88 recharge rate for a particular combination of land use and soil type.
89 
90 **Curve number method.** In the current version of the SWB model, it
91 is
92 assumed that infiltration is the sum of net precipitation, snowmelt,
93 and
94 inflow, minus the runoff calculated by means of the NRCS curve number
95 method. Runoff calculation at a plot or field scale in a continuous
96 simulation by means of the curve number method may be beyond the
97 limits
98 of the method. The list of perceived limitations associated with the
99 curve number method includes the following (Garen and Moore, 2005):
100 
101 - method cannot be used to identify runoff processes, source areas,
102  or
103  flow paths;
104 - method is a watershed-scale method that should not be applied at a
105  plot or field (or grid cell) scale; and
106 - method was developed to evaluate floods and was not designed to
107  simulate daily flows of ordinary magnitude.
108 
109 In addition, it has been suggested that the curve number itself is not
110 constant but varies from event to event and that the antecedent-runoff
111 condition explains only some of this variability
112 (Hjelmfelt, 1991). Given variability in the curve
113 numbers themselves, as well as the other limitations of the curve
114 number
115 method, it is reasonable to treat the standard curve number table
116 values
117 merely as starting points; ideally, the curve numbers should be
118 verified
119 by use of observed paired precipitation-runoff data
120 (Hawkins, 1993).
121 
122 The SWB code contains an alternative method for calculating runoff
123 that
124 incorporates a much smaller initial abstraction term. Use of this
125 alternative method for calculating the initial abstraction may be more
126 appropriate for continuous simulation (Woodward and others, 2003). Users
127 of
128 the SWB code have the option of defining the initial abstraction term
129 as
130 $I_{a} = 0.05S_{max}$, compared to $I_a = 0.2S_{max}$. The use of
131 this
132 smaller initial abstraction term results in more runoff generation for
133 areas with low curve numbers and for storms of smaller magnitude. If
134 the
135 smaller initial abstraction term is used, curve numbers are
136 automatically scaled by the SWB code to maintain an appropriately
137 shaped
138 rainfall-runoff curve (Woodward and others, 2003).
139 
140 The modular design of the SWB code makes it feasible to add new
141 process
142 modules relatively easily. Although there are no immediate plans to do
143 so, future versions of SWB could include an implementation of the
144 Green-Ampt infiltration method (Green and Ampt, 1911), and an enhanced
145 ability to route and store overland flow.
146 
147 **Snowmelt and infiltration.** For temperate areas that experience
148 snowfall and snowmelt, the SWB model is sensitive to snowmelt, and in
149 particular, to how snowmelt translates into surface runoff. The
150 addition
151 of a continuous frozen ground index (CFGI) to the SWB code offers a
152 simple way to approximate the effects of frozen ground. Spring runoff
153 may be increased by lowering the setpoints at which the ground is
154 considered to change from unfrozen to frozen; lowering the CFGI
155 setpoints has the effect of increasing the amount of time that the
156 runoff curve numbers are shifted toward antecedent runoff condition III.
157 
158 Other modelers have altered the curve number in an attempt to simulate
159 runoff from frozen ground. For example, Carroll and others
160 (2005) assigned a separate set of curve numbers to
161 soils considered to be frozen and another set of curve numbers to
162 soils
163 considered to be unfrozen. Despite this, there is no theoretical basis
164 supporting the derivation of a frozen-ground curve number, so its use
165 in
166 the SWB code is primarily for expediency and consistency with other
167 model input considerations.
168 
169 Because the CFGI is based on air temperatures, the SWB code is unable
170 to
171 resolve differences in snowmelt timing between grid cells with
172 differing
173 ground-surface orientation relative to the sun (aspect).
174 
175 **Climate variability.** Year-to-year climate variability causes
176 corresponding variability in calculated recharge values. Use of
177 multiple
178 years of climate data should help to minimize the effect of
179 year-to-year
180 climate variability on estimated recharge values.
