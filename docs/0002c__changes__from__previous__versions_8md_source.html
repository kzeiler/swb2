1 Changes from Previous Versions {#changes_from_previous_versions}
2 ====================================================================
3 
4 The core functionality of SWB has not changed since its initial
5 release
6 (Westenbroek and others, 2010). The code still performs a modified
7 Thornthwaite-Mather soil water balance at each grid point within the
8 model domain. However, the scope of recent process module additions
9 and
10 modifications of the input and output file structures are significant
11 enough to warrant a new major SWB release along with a new publication.
12 
13 Many of the changes to SWB have been "under the hood"; most users will
14 never appreciate the difference between the initial and the current
15 codes. Some of these changes will be very apparent to anyone familiar
16 with the original SWB code. These changes have to do with model input
17 and output, and include:
18 
19 1. Elimination of swbstats,
20 2. Elimination of internally generated graphics,
21 3. Elimination of the custom swb binary output files,
22 4. Addition of PROJ.4 library (allows SWB to read grids with
23  differing
24  geographic projections),
25 5. Upgrading NetCDF input and output to NetCDF version 4,
26 6. Modification of internal structure to make adding new modules
27  easier,
28 7. Addition of facility allowing for more flexible tabular data and
29  parameter input, and
30 8. Rearrangement of internal data structures to more efficiently
31  simulate non-rectangular model domains.
32 
33 Many of the changes outlined above were made in response to
34 frustrations
35 relating to the difficulty of aligning and resampling input grids; SWB
36 1.0 required that every grid supplied to it be in exactly the same
37 geographic projection, cover the same extents as the SWB project grid,
38 at the same gridcell resolution. This requirement resulted in much
39 needless data manipulation and consumed project time that would better
40 have been spent on more worthwhile project tasks. In addition, SWB 1.0
41 stored results in a custom-programmed binary file format. Following a
42 SWB 1.0 model run, a little program called ‘swbstats’ could be run to
43 extract daily, monthly, annual, or period grids as well as generate
44 grids and plots.
45 
46 When the opportunity to modify SWB 1.0 came, the authors decided to
47 overhaul some of the input and output code. SWB 2.0 stores all gridded
48 output in a common gridded and widely used format: NetCDF
49 (Unidata, 2014). The NetCDF file format is in common use amongst
50 climate scientists and meteorologists, and has been slowly gaining
51 favor
52 in other scientific fields. The benefit of switching to a well-known
53 binary file format is that rather than relying on a single program,
54 swbstats, to handle post-processing, there are dozens of active,
55 maintained open-source tools designed to make post-processing of
56 NetCDF
57 files easier.
58 
59 Other changes since the initial SWB release add or modify the actual
60 hydrologic processes simulated by SWB. These changes include the
61 addition of the following modules:
62 
63 1. Irrigation demand,
64 2. FAO-56 soil-moisture retention,
65 3. Interception: a) Horton, b) Gash,
66 4. Fog interception,
67 5. Rainfall: method of fragments,
68 6. Pervious/impervious subgrid characterization, and
69 7. Runoff: monthly runoff ratio.
