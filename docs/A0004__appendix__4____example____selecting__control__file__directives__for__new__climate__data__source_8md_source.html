1 Appendix 4. Example of control file setting selection for a new climate data source.
2 ====================================================================================
3 
4 New sources of gridded climate data are continuing to come online.
5 Inevitably, these new climate data sources will require that the SWB
6 control file directives be modified in order to work properly. This
7 example shows the process used to configure SWB control file
8 parameters
9 for a set of downscaled climate model results produced by a consortium
10 of agencies including USGS, BLM, NCAR, U.S. Army Corps. of Engineers,
11 and others (Brekke and others, 2013), available here:
12 
13 [http://gdo-dcp.ucllnl.org/downscaled\\_cmip\\_projections/dcpInterface.html](http://gdo-dcp.ucllnl.org/downscaled_cmip_projections/dcpInterface.html).
14 
15 For this example, we downloaded model results for a single climate
16 scenario, for a subset of the national grid. As will be shown later,
17 sometimes the tools employed in creating the data subset end up
18 changing
19 the output NetCDF file in a way that prevents it from being used with
20 SWB. Once the necessary NetCDF files have been downloaded to a local
21 hard drive, the metadata they contain should be examined. A simple
22 command-line tool that can accomplish this is called “ncdump”,

99 example, four dimensions are defined: longitude, latitude, time, and
100 projection. In addition, the file contains four variables: longitude,
101 latitude, time, and pr (precipitation). Three of the variable names

135 An interesting this to note about the values of longitude is that they
136 seem unusual relative to the longitudes we are used to working with in
137 North America. Indeed, this example dataset is centered on the state
138 of
139 Nebraska, USA; we commonly would see the longitude values range from
140 about 108° to 93° West longitude, perhaps expressed as -108° to -93°.
141 Many of the downscaled climate model datasets refer to longitude as
142 ranging from 0° to 360°, with the longitude of 0°/360° centered on the
143 parallel running through Greenwich, England. If we subtract 360° from
144 the longitude values above, the range looks more familiar:
145 251°-360°=-108°; 267°-360°=-93°. Presumably the reason for defining
146 longitudes this way is because it is easier to have model grid for
147 which
148 all longitude values are greater than zero!
149 
150 One item we need to look at first is the organization of the data of
151 interest on the disk file. SWB expects climate data files to be
152 arranged
153 in such a way that the data may be accessed by referencing a specific
154 datetime, y-coordinate, and x-coordinate value. The precipitation
155 variable we are interested in is dimensioned as follows:
156 
157  float pr(projection, time, latitude, longitude) ;
158 
159 SWB is written under the assumption that the variable of interest will
160 be referenced by just three dimensions: time, x, and y. The fourth
161 dimension listed above, projection, was added apparently to allow
162 results for more than one climate emissions scenario to be stored in a
163 single NetCDF file. In order to use this file with SWB we must get rid
164 of this fourth dimension. To remove the fourth dimension, we can use a
165 third-party tool called NCO, NetCDF Climate Operators, to calculate an
166 “average” over the fourth dimension. Because there is only a single
167 projection contained in the file, the resulting file will be the same
168 as
169 the input file *without* the projection dimension. NCO as available
170 at:
171 <http://nco.sourceforge.net/>. As of February, 2016, a Windows
172 executable
173 for NCO may be found here:
174 <http://nco.sourceforge.net/src/nco-4.5.4.windows.mvs.exe>.
175 
176 The NCO package is not overly user friendly. Luckily, it has a helpful
177 discussion page, which suggests that to eliminate a “degenerate”
178 dimension (a dimension with only a single value), the “averaging” tool
179 may be used:
180 
181  ncwa -a projection Extraction_pr.nc Extraction_pr_3d.nc
182 
183 ncwa stands for “NetCDF Weighted Averager”. The “-a” flag allows one
184 to
185 specify a dimension over which to average (“projection”). The last two
186 entries are the input and output filenames, respectively. Once this
187 command-line utility is run, the NetCDF files are rendered usable by
188 SWB. With all of the metadata available we can finally generate the
189 SWB
190 control file statements to make this file work with SWB:
191 
192  # 001: specify the filename containing precipitation data
193  PRECIPITATION NETCDF Extraction_pr_3D.nc
194 
195  # 002: define PRECIPITATION projection and NetCDF variable names
196  PRECIPITATION_GRID_PROJECTION_DEFINITION +proj=lonlat +ellps=GRS80 +datum=NAD83 +lon_wrap=180 +no_defs
197  NETCDF_PRECIP_X_VAR longitude
198  NETCDF_PRECIP_Y_VAR latitude
199  NETCDF_PRECIP_Z_VAR pr
200 
201  # 003: define PRECIPITATION missing values action
202  PRECIPITATION_MISSING_VALUES_CODE 1.0E+20
203  PRECIPITATION_MISSING_VALUES_OPERATOR >=
204  PRECIPITATION_MISSING_VALUES_ACTION ZERO
205 
206  # 004: PRECIPITATION is given in mm/day; need to convert to inches/day
207  PRECIPITATION_SCALE_FACTOR 0.03937008
208 
209 The first line of the control file snippet above specifies the name of
210 the NetCDF file that contains the precipitation data we wish to use.
211 
212 The commands under group 002 define the geographic projection and
213 specify the variable names. In this case, the exact projection of the
214 data is unknown; in any event climate models rarely seem to reflect
215 anything other than a global (unprojected) coordinate system. It is
216 not
217 clear from the metadata what datum was used in defining the latitude
218 and
219 longitude. We’ll guess NAD83 with an ellipsoid specified by GRS80.
220 Downscaled climate model cells are generally far larger than the error
221 we would incur by selecting the wrong datum and ellipsoid. Note that
222 this dataset requires the “+lon_wrap=180” addition to the projection
223 definition. This is required in order to convert the longitude values
224 to
225 +/- 180°. The variable names are found in the metadata and must be
226 supplied in the control file in order for SWB to find them: x=&gt;
227 “longitude”, y=&gt;”latitude”, z=”pr”.
228 
229 The commands under group 003 specify what actions SWB should take in
230 the
231 event it encounters missing values within the file. The first line
232 defines the numeric value associated with a missing value. The second
233 line defines the operator, which can be one of “&lt;”, “&lt;=”,
234 “&gt;”,
235 “&gt;=”. The third line specifies what should be done with the missing
236 value. In this case, we’ve specified that any values &gt;= 1.0E+20
237 will
238 be treated as 0.0.
239 
240 The command under group 004 informs SWB how the values in the file are
241 to be converted to inches per day. The PRECIPITATION_SCALE_FACTOR of
242 0.03937 = 1.0 / 25.4, or one over the number of millimeters per inch.
