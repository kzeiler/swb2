25 ! 64-bit KISS RNGs
26 !
27 ! Consistent with the Keep It Simple Stupid (KISS) principle,
28 ! I have previously suggested 32-bit KISS Random Number
29 ! Generators (RNGs) that seem to have been frequently adopted.
30 !
31 ! Having had requests for 64-bit KISSes, and now that
32 ! 64-bit integers are becoming more available, I will
33 ! describe here a 64-bit KISS RNG, with comments on
34 ! implementation for various languages, speed, periods
35 ! and performance after extensive tests of randomness.
36 !
37 ! This 64-bit KISS RNG has three components, each nearly
38 ! good enough to serve alone. The components are:
39 ! Multiply-With-Carry (MWC), period (2^121+2^63-1)
40 ! Xorshift (XSH), period 2^64-1
41 ! Congruential (CNG), period 2^64
42 !
43 ! Compact C and Fortran listings are given below. They
44 ! can be cut, pasted, compiled and run to see if, after
45 ! 100 million calls, results agree with that provided
46 ! by theory, assuming the default seeds.
47 !
48 ! Users may want to put the content in other forms, and,
49 ! for general use, provide means to set the 250 seed bits
50 ! required in the variables x,y,z (64 bits) and c (58 bits)
51 ! that have been given default values in the test versions.
52 !
53 ! The C version uses #define macros to enumerate the few
54 ! instructions that MWC, XSH and CNG require. The KISS
55 ! macro adds MWC+XSH+CNG mod 2^64, so that KISS can be
56 ! inserted at any place in a C program where a random 64-bit
57 ! integer is required.
58 ! Fortran's requirement that integers be signed makes the
59 ! necessary code more complicated, hence a function KISS().
60 !
61 ! C version; test by invoking macro KISS 100 million times
62 ! -----------------------------------------------------------------

70 ! #define XSH ( y^=(y<<13), y^=(y>>17), y^=(y<<43) )
71 ! #define CNG ( z=6906969069LL*z+1234567 )
72 ! #define KISS (MWC+XSH+CNG)

109 !
110 ! Output from using the macro KISS or the function KISS() is
111 ! MWC+XSH+CNG mod 2^64.
112 !
113 ! CNG is easily implemented on machines with 64-bit integers,
114 ! as arithmetic is automatically mod 2^64, whether integers
115 ! are considered signed or unsigned. The CNG statement is
116 ! z=6906969069*z+1234567.
117 ! When I established the lattice structure of congruential
118 ! generators in the 60's, a search produced 69069 as an easy-
119 ! to-remember multiplier with nearly cubic lattices in 2,3,4,5-
120 ! space, so I tried concatenating, using 6906969069 as
121 ! my first test multiplier. Remarkably---a seemingly one in many
122 ! hundreds chance---it turned out to also have excellent lattice
123 ! structure in 2,3,4,5-space, so that's the one chosen.
124 ! (I doubt if lattice structure of CNG has much influence on the
125 ! composite 64-bit KISS produced via MWC+XSH+CNG mod 2^64.)
126 !
127 !
128 ! XSH, the Xorshift component, described in
129 ! www.jstatsoft.org/v08/i14/paper
130 ! uses three invocations of an integer "xor"ed with a shifted
131 ! version of itself.
132 ! The XSH component used for this KISS is, in C notation:
133 ! y^=(y<<13); y^=(y>>17); y^=(y<<43)
134 ! with Fortran equivalents y=ieor(y,ishft(y,13)), etc., although
135 ! this can be effected by a Fortran statement function:
136 ! f(y,k)=ieor(y,ishft(y,k))
137 ! y=f(f(f(y,13),-17),43)
138 ! As with lattice structure, choice of the triple 13,-17,43 is
139 ! probably of no particular importance; any one of the 275 full-
140 ! period triples listed in the above article is likely to provide
141 ! a satisfactory component XSH for the composite MWC+XSH+CNG.
142 !
143 ! The choice of multiplier 'a' for the multiply-with-carry (MWC)
144 ! component of KISS is not so easily made. In effect, a multiply-
145 ! with-carry sequence has a current value x and current "carry" c,
146 ! and from each given x,c a new x,c pair is constructed by forming
147 ! t=a*x+c, then x=t mod b=2^64 and c=floor(t/b).
148 ! This is easily implemented for 32-bit computers that permit
149 ! forming a*t+c in 64 bits, from which the new x is the bottom and
150 ! the new c the top 32-bits.
151 !
152 ! When a,x and c are 64-bits, not many computers seem to have an easy
153 ! way to form t=a*x+c in 128 bits, then extract the top and bottom
154 ! 64-bit segments. For that reason, special choices for 'a' are
155 ! needed among those that satisfy the general requirement that
156 ! p=a*b-1 is a prime for which b=2^64 has order (p-1)/2.
157 !
158 ! My choice---and the only one of this form---is a=2^58+1. Then the
159 ! top 64 bits of an imagined 128-bit t=a*x+c may be obtained as
160 ! (using C notation) (x>>6)+ 1 or 0, depending
161 ! on whether the 64-bit parts of (x<<58)+c+x cause an overflow.
162 ! Since (x<<58)+c cannot itself cause overflow (c will always be <a),
163 ! we get the carry as c=(x>>6) plus overflow from (x<<58)+x.
164 !
165 ! This is easily done in C with unsigned integers, using a different
166 ! kind of 't': t=(x<<58)+c; c=(x>>6); x=t+x; c=c+(x<t);
167 ! For Fortran and others that permit only signed integers, more work
168 ! is needed.
169 ! Equivalent mod 2^64 versions of t=(x<<58)+c and c=(x>>6) are easy,
170 ! and if s(x) represents (x>>63) in C or ishft(x,-66) in Fortran,
171 ! then for signed integers, the new carry c comes from the rule
172 ! if s(x) equals s(t) then c=(x>>6)+s(x) else c=(x>>6)+1-s(x+t)
173 !
174 ! Speed:
175 ! A C version of this KISS RNG takes 18 nanosecs for each
176 ! 64-bit random number on my desktop (Vista) PC, thus
177 ! producing KISSes at a rate exceeding 55 million per second.
178 ! Fortran or other integers-must-be-signed compilers might get
179 ! "only" around 40 million per second.
180 !
181 ! Setting seeds:
182 ! Use of KISS or KISS() as a general 64-bit RNG requires specifying
183 ! 3*64+58=250 bits for seeds, 64 bits each for x,y,z and 58 for c,
184 ! resulting in a composite sequence with period around 2^250.
185 ! The actual period is
186 ! (2^250+2^192+2^64-2^186-2^129)/6 ~= 2^(247.42) or 10^(74.48).
187 ! We "lose" 1+1.58=2.58 bits from maximum possible period, one bit
188 ! because b=2^64, a square, cannot be a primitive root of p=ab-1,
189 ! so the best possible order for b is (p-1)/2.
190 ! The periods of MWC and XSH have gcd 3=2^1.58, so another 1.58
191 ! bits are "lost" from the best possible period we could expect
192 ! from 250 seed bits.
193 !
194 ! Some users may think 250 seed bits are an unreasonable requirement.
195 ! A good seeding procedure might be to assume the default seed
196 ! values then let the user choose none, one, two,..., or all
197 ! of x,y,z, and c to be reseeded.
198 !
199 ! Tests:
200 ! Latest tests in The Diehard Battery, available at
201 ! http://i.cs.hku.hk/~diehard/
202 ! were applied extensively. Those tests that specifically required
203 ! 32-bit integers were applied to the leftmost 32 bits
204 ! (e,g, KISS>>32;), then to the middle 32-bits ((KISS<<16)>>32;)
205 ! then to the rightmost 32 bits, ( (KISS<<32)>>32).
206 ! There were no extremes in the more than 700 p-values returned
207 ! by the tests, nor indeed for similar tests applied to just two of the
208 ! KISS components: MWC+XSH, then MWC+CNG, then XSH+CNG.
209 !
210 ! The simplicity, speed, period around 2^250 and performance on
211 ! tests of randomness---as well as ability to produce exactly
212 ! the same 64-bit patterns, whether considered signed or unsigned
213 ! integers---make this 64-bit KISS well worth considering for
214 ! adoption or adaption to languages other than C or Fortran,
215 ! as has been done for 32-bit KISSes.
216 !
217 ! George Marsaglia

223 ! On Nov 3 I posted

225 ! RNGs: A Super KISS sci.math, comp.lang.c, sci.crypt
226 !
227 ! a KISS (Keep-It-Simple-Stupid) RNG combining,
228 ! by addition mod 2^32, three simple RNGs:
229 ! CMWC(Complementary-Multiply-With-Carry)+CNG(Congruential)
230 ! +XS(Xorshift)
231 ! with resulting period greater than 10^402575.
232 !
233 ! The extreme period comes from finding a prime p=a*b^r+1 for
234 ! which the order of b has magnitude near p-1, then use
235 ! the CMWC method, the mathematics of which I outline here:
236 !
237 ! Let Z be the set of all "vectors" of the form
238 ! [x_1,...,x_r;c] with 0<=x_i<b and 0<=c<a.
239 ! Then Z has ab^r elements, and if the function f() on Z is
240 ! f([x_1,x2,...,x_r;c])=
241 ! [x_2,...,x_r,b-1-(t mod b);trunc((t/b)], t=a*x_1+c
242 ! then f() has an inverse on Z: for each z in Z there is exactly
243 ! one w in Z for which f(w)=z:
244 ! f^{-1}([x_1,x2,...,x_r;c])=
245 ! [trunc((v/a),x_1,...,x_{r-1}; v mod a], v=cb+(b-1)-x_r
246 !
247 ! Thus a directed graph based on z->f(z) will consist only
248 ! of disjoint loops of size s=order(b,p) and there will be
249 ! L=ab^r/s such loops.
250 !
251 ! A random uniform choice of z from Z is equally likely to
252 ! fall in any one of the L loops, and then each "vector" in
253 ! the sequence obtained by iterating f:
254 ! f(z), f(f(z)), f(f(f(z))),...
255 ! will have a uniform distribution over that loop, and the sequence
256 ! determined by taking the r'th element from each "vector",
257 ! (the output of the CMWC RNG) will be periodic with period
258 ! the order of b for the prime p=ab^r+1, and that sequence
259 ! will produce, in reverse order, the base-b expansion of a
260 ! rational k/p for some k determined by choice of the seed z.
261 !
262 ! For that Nov 3 post, I had found that the order of b=2^32
263 ! for the prime p=7010176*b^41790+1 is 54767*2^1337279, about
264 ! 10^402566, thus providing an easily-implemented
265 ! KISS=CMWC+CNG+XS RNG with immense period and
266 ! excellent performance on tests of randomness.
267 !
268 ! That easy implementation required carrying out the essential
269 ! parts of the CMWC function f(): form t=7010176*x+c in 64 bits,
270 ! extract the top 32 bits for the new c, the bottom 32 for
271 ! the new x---easy to do in C, not easy in Fortran.
272 ! And if we want 64-bit random numbers, with B=2^64, our prime
273 ! becomes 7010176*B^20985+1, for which the period of B is
274 ! 54767*2^1337278, still immense, but in C, with 64-bit x,c
275 ! there seems no easy way to form t=7010176*x+c in 128 bits,
276 ! then extract the top and bottom 64 bit halves.
277 !
278 ! So base b=2^32 works for C but not Fortran,
279 ! and base B=2^64 works for neither C nor Fortran.
280 !
281 ! I offer here is a prime that provides CMWC RNGs for both
282 ! 32- and 64-bits, and for both C and Fortran, and with
283 ! equally massive periods, again greater than 2^(1.3million):
284 !
285 ! p=640*b^41265+1 = 2748779069440*B^20632+1 = 5*2^1320487+1.
286 !
287 ! That prime came from the many who have dedicated their
288 ! efforts and computer time to prime searches. After some
289 ! three weeks of dedicated computer time using pfgw with
290 ! scrypt, I found the orders of b and B:
291 ! 5*2^1320481 for b=2^32, 5*2^1320480 for B=2^64.
292 !
293 ! It is the choice of the "a" that makes it feasible to get
294 ! the top and bottom valves of t=a*x+c, yet stay within the
295 ! integer sizes the C or Fortran compilers are set for.
296 ! In the above prime: a=640=2^9+2^7 for b=2^32 and
297 ! a=2748779069440=2^41+2^39 for B=2^64.
298 ! Thus, for example with b=2^32 and using only 32-bit C code,
299 ! with a supposed 128-bit t=(2^9+2^7)*x+c, the top and bottom
300 ! 32-bits of t may be obtained by setting, say,
301 ! h=(c&1); z=(x<<9)>>1 + (x<<7)>>1 + c>>1;
302 ! then the top half of that t would be
303 ! c=(x>>23)+(x>>25)+(z>>31);
304 ! and the bottom half, before being complemented, would be
305 ! x=(z<<1)+h;
306 !
307 ! When B=2^64 we need only change to
308 ! h=(c&1); z=(x<<41)>>1 + (x<<39)>>1 + c>>1;
309 ! c=(x>>23)+(x>>25)+(z>>63);
310 !
311 ! These C operations all have Fortran equivalents, and will
312 ! produce the required bit patterns, whether integers are
313 ! considered signed or unsigned. (In C, one must make sure
314 ! that the >> operation performs a logical right shift,
315 ! perhaps best done via "unsigned" declarations.)
316 !
317 ! The CMWC z "vector" elements [x_1,x_2,...,x_r] are kept in
318 ! an array, Q[] in C, Q() in Fortran, with a separate current
319 ! "carry". This is all spelled out in the following examples:
320 ! code for 32- and 64-bit SuperKiss RNGs for C and Fortran.
321 !
322 ! Note that in these sample listings, the Q array is seeded
323 ! by CNG+XS, based on the seed values specified in the
324 ! initial declarations. For many simulation studies, the
325 ! 73 bits needed to seed the initial xcng, xs and carry<a
326 ! for the 32-bit version, or 169 bits needed for the 64-bit
327 ! version, may be adequate.
328 ! But more demanding applications may require a significant
329 ! portion of the >1.3 million seed bits that Q requires.
330 ! See text and comments from the Nov 3 posting.
331 !
332 ! I am indebted to an anonymous mecej4 for providing the basic
333 ! form and KIND declarations of the Fortran versions.
334 !
335 ! Please let me and other readers know if the results are not
336 ! as specified when run with your compilers, or if you can
337 ! provide equivalent versions in other programming languages.
