2 -------------------------------------------------------------------------------------
3 
4 SWB can ingest gridded data in three formats: Surfer, ESRI Arc ASCII,
5 or
6 netCDF.

8 Supported File Types
9 ====================

11 Surfer ASCII Grid
12 =================
13 
14 Golden Software’s ASCII grid format consists of a 5-line header
15 followed
16 by the data values arranged in a matrix.
17 
18  DSAA
19  14 5
20  0.5 7.0
21  -0.4 0.0
22  0.0 7.0
23  0.5 1. 1.5 2. 2.5 3. 3.5 4. 4.5 5. 5.5 6. 6.5 7.0
24  0.45 0.9 1.4 1.9 2.4 2.9 3.4 3.9 4.4 4.9 5.4 5.9 6.4 6.9
25  0.4 0.8 1.3 1.8 2.3 2.8 3.3 3.8 4.3 4.8 5.3 5.8 6.3 6.8
26  0.36 0.72 1.21 1.7 2.2 2.7 3.2 3.7 4.2 4.7 5.2 5.7 6.2 6.7
27  0.32 0.64 1.13 1.6 2.1 2.6 3.1 3.6 4.1 4.6 5.1 5.6 6.1 6.6
28 
29 The header values contain:
30 
31 1. “DSAA”, a label identifying the file format as a Golden Software
32  ASCII grid,
33 2. Number of columns (number of X values), number of rows (number of
34  Y
35  values),
36 3. Minimum X value, maximum X value,
37 4. Minimum Y value, maximum Y value,
38 5. Minimum Z value, maximum Z value.
39 
40 As one might expect, the matrix of values is arranged such that the
41 “x”
42 coordinates increase from lower to higher values as one moves left to
43 right over the columns. The “y” coordinates correspond to the rows of
44 data and decrease from higher values to lower values as one moves down
45 the rows.

47 ESRI ASCII Grid
48 ===============
49 
50 The publishers of ArcMap and ArcView software, the Environmental
51 Systems
52 Research Institute (ESRI), developed one of the most commonly used
53 raster data formats in use. ESRI’s ASCII grid format is a simple
54 matrix
55 representation of the gridded dataset with a short header tacked to
56 the
57 top of the file.

77 NetCDF
78 ======
79 
80 NetCDF, or Network Common Data File, is a file format commonly used by
81 researchers in atmospheric and oceanic sciences. One of the key
82 benefits
83 of NetCDF files is that they are designed to be platform-independent;
84 in
85 other words, a NetCDF file generated on a Macintosh by an application
86 compiled with the PGI compiler should be able to be read by an
87 application compiled with the Intel compiler and running on Windows.
88 In
89 addition, NetCDF files are able to store arbitrary combinations of
90 data.
91 This allows for substantial metadata to be stored in the NetCDF file
92 along with the variable of interest.
93 
94 A set of conventions, known as the Climate and Forecast Metadata
95 Conventions, gives recommendations regarding the kind and nature of
96 metadata to be included along with the primary variable within a
97 NetCDF
98 file (Eaton and others, 2011). SWB outputs written to NetCDF files
99 attempt
100 to adhere to the Climate and Forecast Metadata Conventions version 1.6
101 (CF 1.6) in order to maximize the number of third-party NetCDF tools
102 that will work with SWB output.
103 
104 In addition to these benefits of NetCDF file use, the fact that there
105 are dozens of open-source tools available to read, write, and
106 visualize
107 NetCDF files makes them a good candidate for use with SWB. One of the
108 most basic tools, distributed by Unidata, the maintainer of NetCDF
109 file
110 format, is called ncdump—a program to “dump” the contents of a NetCDF
111 file. If we take a recent Daymet file containing daily precipitation
112 values for 2014, we can see the stored metadata in the file:
113 
114  > ncdump -h prcp_2014.nc4
115  The following metadata is returned:

172  :references = "Please see http://daymet.ornl.gov/ for current information on Daymet references" ;
173  }
174 
175 This particular file contains three classes of metadata: dimensions,
176 variables, and global attributes. As can be seen above, the file
177 contains data about 4 “dimensions”: x, y, time, and nv. Nine variables
178 are defined, each of which is references in terms of the dimensions.
179 The
180 key variable in the file is “prcp”—the daily precipitation value. prcp
181 is defined at each time (day) in the file for all values of x and y.
182 Note the way that dates and times are specified in the NetCDF file: as
183 a
184 real-valued number of days since 1980-01-01 00:00:00 UTC.
185 
186 SWB does not have the ability to make sense of much of this metadata.
187 It
188 is the user’s responsibility to be aware of the physical units that
189 each
190 of the datasets is stored in. Control file directives may be used to
191 convert precipitation in metric units (mm/day) to inches per day. We
192 recommend examining the output values of air temperature and
193 precipitation in order to verify that any such unit conversions have
194 been done correctly.
195 
196 In addition, SWB cannot parse the variables and attributes associated
197 with any map projection that may have been used when the NetCDF file
198 was
199 created. The user needs to be aware of the geographic projection (if
200 any) that was used. If the gridded data do not match the SWB project
201 bounds exactly, a “PROJ4 string” must be provided to enable SWB to
202 translate between project coordinates and the NetCDF file coordinates.
203 
204 Geographic Projections
205 ======================
206 
207 Later versions of SWB are linked to the PROJ4 library, originally
208 written by a USGS scientist (Evenden, 1990). PROJ4 is used
209 by many commonly used mapping applications, both open source and
210 proprietary. The library provides a set of routines that may be used
211 to
212 calculate forward and backward coordinate transformations. SWB make
213 use
214 of this library in order to translate coordinates found in external
215 data
216 files to the coordinates in use by the SWB project.
